# context-compression

这是一个用于“上下文压缩”的本地 skill，目标是在对话变长、日志变多、代码改动累积、准备切换任务或准备交接时，把当前状态压缩成一个可继续执行的短摘要。

## 这个 skill 是做什么的

- 在长对话中提炼当前目标、关键决策、约束、验证状态和下一步动作
- 减少重复复述和低价值上下文
- 为下一个回合、下一个代理或稍后恢复工作提供 continuation brief

## 目录说明

- `SKILL.md`
  - 主规则文件
  - 定义触发时机、压缩目标、保留字段、输出模板和质量检查
- `examples.md`
  - 配套示例文件
  - 通过正反例帮助模型识别何时该压缩、怎样压缩得更稳

## 两个文件为什么要配合使用

这两个文件是组合设计：

- `SKILL.md` 负责定义规则
- `examples.md` 负责提供样例

这样拆开有两个直接好处：

1. 主规则更短、更稳定，便于被优先加载和复用
2. 示例外置，只有在需要增强判断或校准输出风格时再参考

## 和“增强缓存命中”的关系

如果你说的“命中缓存”是指 prompt/context cache，那么这套拆分方式是有帮助的。

原因是：

- 稳定且短小的主文件更容易保持一致
- 大段示例不必每次都和主规则一起加载
- 规则层和示例层分离后，更利于复用稳定前缀

所以更准确地说：

- `SKILL.md` 是核心规则载体
- `examples.md` 更像是为了增强触发识别和输出一致性，也间接有利于缓存命中

## 适用场景

- 长实现线程
- 大量日志排查
- 多文件修改后准备切换任务
- 会话即将结束，需要交接或稍后恢复
- token 紧张时需要保留高价值上下文

## 缺点与局限

这个 skill 有用，但它并不是“无损压缩”，也不适合所有场景。

- 信息损耗不可避免
  - 一旦进入压缩，细枝末节、边缘分支和一些当下看起来“不重要”的上下文可能被折叠掉，后续如果这些细节重新变重要，就需要回头重新查原始对话、日志或代码。
- 容易把错误判断也一起压缩进去
  - 如果压缩前的理解本身就偏了，压缩后的 brief 可能会把这种偏差“固化”，让后续代理更容易在错误前提上继续推进。
- 对压缩时机有要求
  - 压得太早，可能丢掉还没完成验证的信息；压得太晚，已经消耗了大量上下文预算，收益会变差。
- 不能替代真实验证
  - 它可以保留“验证状态”，但不能替代测试、lint、运行结果或 diff 检查；压缩摘要本身不是证据。
- 对简单任务反而可能增加负担
  - 如果任务很短、改动很小、上下文很清晰，额外做一次压缩会增加步骤，收益不一定高。
- 输出质量依赖执行者判断
  - 哪些信息算“高价值”、哪些应该删掉，仍然依赖代理当下的判断；不同执行者压出来的结果可能不完全一致。

所以更准确地说，`context-compression` 更像是：

- 一个降低上下文膨胀风险的工作流工具
- 不是严格意义上的事实存档系统
- 也不是交付验证、变更审计或知识库沉淀的替代品

## 不适合的场景

- 只有单一步骤的小任务
- 还在探索根因、结论频繁变化的早期排查阶段
- 需要逐字保留原始日志、报错、命令输出的审计型场景
- 需要完整历史链路而不是摘要的复盘型场景

## 维护建议

- 优先保持 `SKILL.md` 稳定，不要频繁改动结构
- 新增案例时尽量放到 `examples.md`
- 示例应覆盖好输出、坏输出、不同复杂度和不同场景
- 如果后续需要面向中文团队优化，可继续追加中文示例，但尽量不要破坏主模板结构
