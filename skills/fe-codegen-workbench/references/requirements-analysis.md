# 需求分析与结构化提取

本文件为 **fe-codegen-workbench** 流程第 2 步的详细说明。

## 分析流程

```
用户输入
  ├─ 纯文本需求 → 文本解析
  ├─ 原型图/设计稿 → 图片分析 + 文本补充
  └─ 接口文件/类型定义 → 字段提取
      ↓
结构化需求清单
```

## 1. 文本需求解析

从用户文本中提取以下信息：

| 提取项 | 识别关键词 | 示例 |
|--------|-----------|------|
| 页面类型 | 列表、表单、详情、弹窗、抽屉 | "做一个列表页" → 列表页 |
| 功能点 | 搜索、新增、编辑、删除、导入、导出 | "支持搜索和批量删除" |
| 字段列表 | 用户提供的接口结构或字段描述 | {"name": "姓名"} |
| 业务规则 | 默认值、限制、校验、权限 | "文件不超过5MB" |
| 交互方式 | 弹窗/抽屉/跳转新页面 | "点击查看跳转详情页" |
| 页面数量 | 用户描述的页面个数 | "包含列表、详情、编辑三个页面" |
| 参考页面 | 参考、对齐、沿用、仿照 | "参考 src/pages/user/whitelist 的代码风格" |

## 2. 原型图分析

当用户提供原型图/设计稿/截图时：

### 分析要点

1. **布局结构**：页面整体布局（上下/左右/卡片/Tab）
2. **组件识别**：表格、表单、按钮、搜索栏、标签页、弹窗
3. **字段提取**：表头名称、表单标签、输入框类型
4. **交互线索**：按钮文字、操作列内容、状态标签
5. **样式特征**：颜色、间距、字体大小（如需还原设计稿）

### 分析输出模板

```
从原型图分析得出：
- 页面类型: 标准列表页（搜索 + 表格 + 操作）
- 搜索区域: 3 个搜索条件（姓名-输入框, 状态-下拉, 日期-范围选择）
- 表格字段: 序号, 姓名, 工号, 部门, 状态, 创建时间, 操作
- 操作列: 查看, 编辑, 删除
- 工具栏: 新增按钮, 批量导出按钮
- 弹窗/跳转: 点击编辑打开弹窗表单
```

## 3. 字段智能分类

根据字段名称/类型自动判断组件和搜索方式：

| 字段特征 | 判定类型 | 表格展示 | 搜索组件 | 表单组件 |
|----------|----------|----------|----------|----------|
| id/序号 | ID | 序号列 | - | - |
| name/名称/标题 | 文本 | 直接展示 | Input | Input |
| code/编号 | 文本 | 直接展示 | Input | Input |
| status/type/状态/类型 | 枚举 | Tag 标签 | Select | Select/Radio |
| amount/price/数量 | 数字 | 数字展示 | InputNumber 范围 | InputNumber |
| createTime/date/日期 | 日期 | 格式化展示 | DateRangePicker | DatePicker |
| isValid/enabled | 布尔 | Switch/Tag | Select | Switch |
| description/备注 | 长文本 | Tooltip 展示 | - | TextArea |
| departmentId/userId | 关联 | 关联名称 | 远程搜索 | Select 远程 |
| file/attachment/附件 | 文件 | 链接/预览 | - | Upload |

## 4. 页面类型判断

### 模板匹配规则

| 需求特征 | 匹配模板 | 关键标识 |
|----------|----------|----------|
| 搜索+表格+CRUD | 标准列表页 | "列表"/"表格"/"增删改查" |
| 新增/编辑弹窗（<10字段） | 弹窗表单 | "弹窗"/"对话框" |
| 新增/编辑抽屉（10-20字段） | 抽屉表单 | "抽屉"/"侧边" |
| 独立新增/编辑页面（>20字段） | 独立表单页 | "跳转"/"新页面"/"独立" |
| 查看详情-快速预览 | 抽屉详情 | "快速查看"/"侧边详情" |
| 查看详情-完整信息 | 独立详情页 | "详情页"/"审批流程" |
| Excel 数据导入 | 导入弹窗 | "导入"/"excel"/"xlsx" |
| 文件/图片上传 | 上传组件 | "上传"/"附件"/"图片" |
| 虚拟列表/大数据 | 大数据渲染 | "虚拟列表"/"8000条" |

### 组合匹配规则

当需求同时包含主页面和增强能力时，输出“1 个主模板 + 0-N 个组合模板”：

| 需求组合 | 主模板 | 组合模板 |
|---------|-------|---------|
| 列表 + 导入 | `*-standard-list-crud` | `*-import-list-modal` |
| 列表 + 批量编辑 | `*-standard-list-crud` | `react-batch-schema-form` |
| 表单 + 上传 | `*-standard-form-page` / `*-drawer-form` | `*-pc-file-upload` / `vue2-h5-file-upload` |

### 接口数据结构自动判断

默认不要求用户显式声明“文件模式”或“JSON 模式”，按输入内容自动判断：

| 输入信号 | 处理方式 |
|---------|---------|
| 出现 `接口类型：xxx.d.ts`、`接口：xxxApi` | 进入引用已有接口文件模式 |
| 提供 JSON 结构 | 进入 JSON 骨架生成模式 |
| 两者同时提供 | 以接口文件为主，JSON 作为补充字段 |
| 两者都没有，但给了原型图/字段描述 | 从需求中提取字段并生成最小骨架 |

### 多页面拆分

当需求涉及多个页面时，拆分为独立的需求单元。

## 5. 设计风格信号抽取（步骤 2.5 依据）

从用户需求中额外抽取"设计风格信号"，作为步骤 2.5（设计推荐）的触发依据。**即便用户不提风格词，也要完成此项抽取，抽取结果可为空。**

### 5.1 风格信号类别

| 信号类别 | 识别关键词 | 示例 |
|---------|-----------|------|
| **显式品牌名** | linear / notion / stripe / vercel / cursor / claude / supabase / airbnb / shopify 等 index.json 中的品牌名 | "用 linear 风格做个看板" |
| **本地 DESIGN.md 路径** | "设计参考 xxx.md"、"按 xxx/DESIGN.md" | "设计参考 ./my-brand.md" |
| **自然语言风格词** | 暗色/dark、极简/minimal、高级/premium、年轻/playful、编辑感/editorial、数据密集/dense、渐变/neon、赛博/朋克、冷淡/克制 等 | "做得年轻一点 / 要点赛博感" |
| **明确走默认** | "B 端标准"、"按默认"、"按组件库风格"、"别搞花里胡哨的" | "就是普通后台，按默认风格" |

### 5.2 输出字段（追加到步骤 2 结构化需求里）

```json
{
  "designSignal": {
    "explicitBrandId": "<index.json 中的 brand id，或 null>",
    "localDesignMdPath": "<本地路径，或 null>",
    "styleKeywords": ["暗色", "数据密集"],
    "forceDefault": false,
    "productCategory": "b2b-admin | consumer | dashboard | docs | h5 | landing | ai-product"
  }
}
```

### 5.3 设计推荐触发规则（供步骤 2.5 使用）

**默认策略**：新建任务（非增量修改） + `forceDefault = false` → **默认进入 2.5 推荐节点**，给用户一次 3+1 的交互选择机会。以下是细化分支：

| 字段状态 | 步骤 2.5 行为 |
|----------|---------------|
| `explicitBrandId` 非空 | **强触发·跳过推荐对话**，直接用该品牌进入 3.5 |
| `localDesignMdPath` 非空 | **强触发·跳过推荐对话 + 跳过远程抓取**，直接读本地文件进入 3.5 |
| `forceDefault = true` | **显式跳过**，记录 `selectedBrandId = null`，直接进入步骤 3 |
| **增量修改任务**（主 SKILL 增量修改模式触发） | **默认跳过** 2.5，沿用现有主题；除非用户同时明说换主题 |
| `styleKeywords` 非空 + 上述均不命中 | **主动推荐（强匹配）**，按风格词语义从 `brands[]` 选 Top 3 + "不使用"兜底 |
| `styleKeywords` 为空 + 新建任务 | **主动推荐（通用匹配）**，按 `productCategory` 选 3 个普适品牌 + "不使用"兜底，对 `b2b-admin` 类推荐时 `defaultOption` 置顶引导 |

**兜底选项永远存在**：无论进入 2.5 的是"强匹配"还是"通用匹配"，ask_question 都必须包含第 4 项 _"不使用品牌（使用默认 Ant Design 样式）"_，用户一秒即可退出。

**超时处理**：ask_question 无响应 / 用户回复"算了/跳过" → 视同选第 4 项（`selectedBrandId = null`），继续流程不打扰。

## 6. 输出格式

分析完成后输出结构化需求清单，至少包含：

- `primaryTemplateId`
- `composedTemplateIds`
- `referencePages`
- `dataSourceMode`（`existing-interface` / `json-schema` / `hybrid` / `prototype-derived`）
- 字段、业务规则、交互方式
- `designSignal`（第 5 节结构）

## 需要用户确认的决策点

1. 页面类型不确定（弹窗 vs 新页面）
2. 字段不完整
3. 业务规则模糊
4. 设计稿缺失细节
5. `designSignal.styleKeywords` 非空但无 explicitBrandId → 是否进入步骤 2.5 设计推荐
